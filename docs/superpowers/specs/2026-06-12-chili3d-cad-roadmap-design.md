# Chili3D CAD Feature Roadmap — Design / Gap Analysis

**Date:** 2026-06-12
**Status:** Approved roadmap (decomposition + sequencing). Each sub-project below gets its own spec → plan → build cycle.
**Scope:** Identify every significant missing CAD feature in Chili3D and sequence the work to add them, ordered by dependency and risk.

---

## 1. Purpose

This is a **roadmap**, not a single implementation spec. It captures the verified current-state architecture, the full gap analysis split into three cost tiers, and a dependency-ordered build sequence. "All of A→C" was the agreed scope; because that spans ~16 independent subsystems, it is decomposed here into sub-projects. Each sub-project will be brainstormed and specified individually before implementation.

All findings below are grounded in the codebase (file:line references), not benchmarked from memory.

---

## 2. Verified current state

Chili3D is a browser-based parametric CAD app: an OCCT (OpenCASCADE) v8.0.0 kernel compiled to WebAssembly, rendered with Three.js, on a TypeScript npm-workspace monorepo.

### 2.1 What exists today

- **Primitives:** box, sphere, cylinder, cone, pyramid (`cpp/src/factory.cpp:56-188`)
- **2D / curves:** line, arc (2-pt/3-pt), circle, ellipse, polygon, regular polygon, rect, bezier, point, wire, face
- **3D modeling:** extrude (prism), revolve, sweep, loft, pipe, offset, thickSolid (shell), section, boolean (cut/fuse/common)
- **Modify:** fillet, chamfer, move, rotate, mirror, array, trim, split, break, sew, simplify, explode, remove-feature/sub-shapes, brush
- **Measure:** length, angle, selection info
- **Interop:** STEP, IGES, BREP, STL, PLY, OBJ (import/export; OBJ export-only via Three.js)
- **Infrastructure:** parametric body nodes, snapping + dynamic working plane, undo/redo (50-deep, in-memory), IndexedDB persistence, component instancing, plugin system

### 2.2 Architecture facts that drive the roadmap

1. **Parametric per-feature, but geometry-snapshot document.** Each body node extends `ParameterShapeNode` (`packages/core/src/model/shapeNode.ts:168-201`); changing a `@property` calls `setPropertyEmitShapeChanged` → `generateShape()` and regenerates that node's geometry.
2. **No inter-feature dependency graph (keystone gap).** Features store the resulting `IShape` *directly* (e.g. `ExtrudeNode.section` is an `IShape`, `packages/app/src/bodys/extrude.ts:24-64`), not a reference to a source feature. Modify commands are **destructive**: Fillet creates a new `EditableShapeNode` and deletes the original (`packages/app/src/commands/modify/fillet.ts:31-46`); Boolean removes its source nodes. Editing a base feature does **not** rebuild dependents — they no longer exist. The document is a parameter+geometry snapshot, not a parameter recipe.
3. **No constraint solver / no Sketch container.** 2D entities are independent parametric nodes with no relations. No constraint/solver/DoF code anywhere (`packages/core/src/snap/` provides snapping only; snap "perpendicular/parallel" are one-time spatial helpers, not persistent constraints). OCCT does not provide a 2D constraint solver.
4. **No global parameters / expression engine.** No named parameters, formulas, or expression evaluator anywhere in the codebase.
5. **History is in-memory only.** `History` (`packages/core/src/foundation/history.ts:13-102`), capped at 50, disabled during deserialization — not persisted with the document.
6. **Node tree already supports assemblies/kinematics scaffolding.** `ComponentNode` → shared `Component` definition gives part instancing (`packages/core/src/model/component.ts:79-314`, used by Array). Per-node `Matrix4` transform (`visualNode.ts:12-20`) with automatic parent×child world-transform composition via Three.js (`threeVisualObject.ts:87-88`).
7. **Clean, low-boilerplate extension seams.** New command/primitive ≈ 6 files / ~250 LoC. `@command` + `@property` + `@serializable`/`@serialize` decorators auto-wire dispatch, property UI, and persistence. Ribbon layout in `packages/builder/src/ribbon.ts`. Plugin system can add commands/ribbons/i18n/services, but **not** new serializable node types easily.
8. **C++ binding pattern is consistent.** Adding a bound factory op ≈ 20-40 lines C++ + 10-15 lines TS; `.d.ts` auto-generated by Emscripten (`cpp/src/factory.cpp:522-539` fillet is the reference pattern).

## 3. Gap analysis — three cost tiers

### 🟢 Tier A — "Not missing, just unexposed" (days each, zero architecture change)

OCCT v8 already implements these; they are compiled into the WASM and only lack a binding + command.

| Feature | OCCT class (already in build) | Est. |
|---|---|---|
| Hole (counterbore/countersink) | `BRepFeat_MakeCylindricalHole` | 2-4 h |
| Draft angle | `BRepOffsetAPI_DraftAngle` | 2-3 h |
| Variable-radius fillet | `BRepFilletAPI_MakeFillet` (bound single-radius only) | 1-2 h |
| Rib / slot | `BRepFeat_RibSlot` | 4-6 h |
| Pipe feature | `BRepFeat_MakePipe` | 2-3 h |
| Surface fill / patch | `BRepOffsetAPI_MakeFilling` | 4-8 h |
| Thread (helix + existing sweep) | helix curve + bound `sweep()` | 3-5 h |
| Interop: glTF / 3MF / DXF export | Three.js exporters / new converter path | varies |

Each is its own mini sub-project sharing one pattern (C++ bind → TS wrapper → body node → create command → ribbon entry → i18n). Zero architectural risk.

### 🟠 Tier B — Robotics / assembly layer (weeks, additive — no core rewrite)

The node tree already provides component instancing, per-node transforms, and automatic world-transform composition. A kinematic chain maps on as a **new additive abstraction**, not a rewrite.

- **B1 — Link/Joint nodes:** subclass `VisualNode` into `LinkNode` (mass/inertia) and `JointNode` (type: revolute/prismatic/fixed/floating, axis, limits, value). Joint transform computed from its DOF value.
- **B2 — Forward kinematics:** evaluate joint values → link world transforms (rides existing Three.js composition).
- **B3 — URDF import/export:** new converter alongside STEP/IGES (`packages/wasm/src/converter.ts`); parse `<link>`/`<joint>`, export visual+collision meshes per link.
- **B4 — glTF export:** shared with Tier A interop.

Serialization, transforms, and visuals already handle any `VisualNode` subclass, so this is additive. What is genuinely missing is only joint *semantics* and FK evaluation, not the scaffolding.

### 🔴 Tier C — Parametric bedrock (months, sequenced — the true foundations)

Strict internal order; each depends on the prior.

- **C1 — Feature-dependency graph + rebuild engine (keystone).** Convert destructive modifies to referential, non-destructive features; store feature references instead of baked `IShape`; build a DAG and a rebuild traversal so editing an upstream feature rebuilds dependents. Requires serialization-version bump + migration.
- **C2 — Editable feature timeline.** UI + persistence on top of C1 (reorder, suppress, roll-back-edit-roll-forward).
- **C3 — Global parameters / expression engine.** Named parameters + formula evaluator; needs C1's graph to propagate changes across features.
- **C4 — Constraint sketcher.** `SketchNode` container + 2D constraint solver (integrate/port PlaneGCS- or SolveSpace-style; the one piece OCCT cannot provide) + DoF tracking + constraint UI. Largest single effort (~13-18 weeks).

## 4. Build sequence

**Ordering claims:** Tier A and Tier B are each independent of Tier C (quick wins and robotics ship without waiting on the rearchitecture). Within Tier C, C1 (dependency graph) is the hard prerequisite for C2, C3, and a truly parametric C4.

### Phase 0 — De-risk the keystone (spike, ~1 week)
Throwaway spike proving a feature-dependency rebuild graph can coexist with the current destructive-modify model. Decides whether Tier C is "retrofit" or "rewrite." Cheap insurance before any Tier C commitment.

### Phase 1 — Tier A: Manufacturing features (parallelizable, ~3-5 weeks)
Hole → variable fillet → draft → pipe → rib/slot → surface fill → thread; plus glTF/3MF/DXF export. Exercises the full C++→command→ribbon pipeline, ships visible wins, builds momentum at zero architectural risk.

### Phase 2 — Tier B: Robotics/assembly layer (~4-8 weeks)
B1 Link/Joint nodes → B2 forward kinematics → B3 URDF I/O (B4 glTF done in Phase 1). Independent of Tier C; highest value-per-effort for the robotics context.

### Phase 3 — Tier C: Parametric bedrock (months, sequenced)
C1 dependency graph + rebuild engine → C2 editable timeline → C3 parameters/expressions → C4 constraint sketcher.

### Cross-cutting (every sub-project)
Rstest coverage; i18n keys (`en`, `zh-cn`, `pt-br`); ribbon placement; Biome/clang-format. Tier C items each need a serialization-version bump + migration path.

## 6. Risks & open questions

- **C1 is the program's pivot.** If retrofitting a dependency graph onto the destructive-modify model proves infeasible (P0 outcome), Tier C becomes a larger rewrite and may warrant a forked document model. P0 exists to surface this early.
- **2D solver sourcing (C4).** Port vs. integrate an existing solver (PlaneGCS, SolveSpace) vs. write one — to be decided in C4's own spec; licensing matters (project is AGPL-3.0 TS / LGPL-3.0 C++).
- **Plugin vs. core.** New serializable node types (Link/Joint, Sketch) cannot live cleanly in the runtime plugin system today; Tier B/C land in core. Tier A commands *could* be plugin-delivered but are assumed in-core for cohesion.
- **Serialization/version migration.** Tier C changes the document model; a migration strategy (and the existing hard version-mismatch rejection in `document.ts`) must be addressed per sub-project.
- **No feature-tree today means no regression baseline for parametric edits.** Tier C needs new test fixtures, not just unit tests.
